النقاط المخفية والتريكايات - Data Link Layer

أسئلة صعبة ونقاط ما بين السطور مش موجودة في الشرح العادي

النقطة المخفية 1: ليه قالوا "glued together"؟
ما بين السطور:
السلايد قال "The Internet is a combination of networks glued together"

المعنى المخفي:
- كلمة "glued" معناها "ملزوقين" مش متوصلين عادي
- ده معناه إن الشبكات دي أصلاً مستقلة وكل واحدة ليها تكنولوجيا مختلفة
- Routers والـ Switches هما الـ "لاصق" (glue) اللي بيوحد الشبكات دي
- يعني ممكن يكون عندك Ethernet في مكان، WiFi في مكان، Fiber في مكان تالت
- كلهم بيتوصلوا مع بعض رغم اختلافهم
التريكاية:
لو جالك سؤال: "ليه الإنترنت محتاج Routers؟"
الإجابة الغلط: "علشان يوصل الشبكات ببعض" (ده سطحي)
الإجابة الصح: "علشان يربط شبكات مختلفة التكنولوجيا ويخليها تفهم بعض ويعمل translation بين البروتوكولات المختلفة"
1
إيه الفرق بين Switch و Router في سياق "gluing networks"؟
الإجابة:
- Switch: بيربط أجهزة على نفس الشبكة (Same network, same technology)
- Router: بيربط شبكات مختلفة (Different networks, can be different technologies)
- Switch شغال على Layer 2 (MAC Addresses)
- Router شغال على Layer 3 (IP Addresses)
2
لو عندك شبكة Ethernet وشبكة WiFi، مين المسؤول عن ربطهم؟ ليه؟
الإجابة:
Router (أو Access Point بوظيفة Router)
لأن دول تكنولوجيتين مختلفتين محتاجين Layer 3 routing علشان يتكلموا
3
هل ممكن يكون في إنترنت من غير Routers؟ ليه؟
الإجابة:
لا، مستحيل
- الإنترنت بطبيعته عبارة عن شبكات متعددة
- محتاجين Routers علشان ينقلوا الـ Packets بين الشبكات دي
- من غير Routers كل شبكة هتفضل معزولة
النقطة المخفية 2: Node-to-Node vs End-to-End
ما بين السطور:
السلايد قال "Communication at the data-link layer is node-to-node"

المعنى المخفي:
- Data Link Layer مش شايفة الصورة الكبيرة خالص
- كل اللي يهمها الجهاز اللي قدامها مباشرة
- مش عارفة حاجة عن المصدر الأصلي أو الوجهة النهائية
- ده معناه إن الـ Frame بيتعمل rebuild في كل Router
- MAC Addresses بتتغير في كل hop، لكن IP بيفضل ثابت
التريكاية الخطيرة:
سؤال: "هل MAC Address بيتغير أثناء نقل البيانات؟"
- لو قصدك الـ Frame نفسه: أيوه بيتغير في كل hop
- لو قصدك MAC Address بتاع الجهاز نفسه: لا طبعاً ثابت
الطلبة بتلخبط في السؤال ده جداً!
4
لو Alice بعتت Frame لـ Bob عبر 3 Routers، كام Frame هيتعمل في الرحلة؟
الإجابة:
4 Frames مختلفين!
- Frame 1: من Alice لـ R1
- Frame 2: من R1 لـ R2
- Frame 3: من R2 لـ R3
- Frame 4: من R3 لـ Bob
كل واحد له MAC Addresses مختلفة
5
إيه اللي بيفضل ثابت في الرحلة من Alice لـ Bob؟ وإيه اللي بيتغير؟
الإجابة:
ثابت:
- IP Source (Alice)
- IP Destination (Bob)
- الـ Data نفسه

بيتغير:
- Source MAC Address
- Destination MAC Address
- الـ Frame Header كله
- الـ FCS (Frame Check Sequence)
6
ليه Data Link Layer مابتشوفش الصورة الكبيرة؟ إيه الحكمة من ده؟
الإجابة:
- Separation of Concerns: كل طبقة تركز في شغلها بس
- Scalability: لو Data Link عرفت كل حاجة هتبقى معقدة جداً
- Flexibility: نقدر نغير التكنولوجيا في كل link من غير ما نأثر على الباقي
- Efficiency: كل hop يهتم بس بتوصيل الـ Frame للـ hop التالي بنجاح
النقطة المخفية 3: الفرق بين Node و Host
ما بين السطور:
السلايد قال "refer to the two end hosts and the routers as nodes"

المعنى المخفي:
- Host = جهاز في البداية أو النهاية (End device)
- Node = أي جهاز في المسار (Host أو Router)
- يعني كل Host هو Node، لكن مش كل Node هو Host
- الـ Router يعتبر Node لكن مش Host
- Switch ممكن يتعتبر Node في بعض السياقات
التريكاية:
سؤال امتحان شهير: "هل الـ Router يعتبر Host؟"
- الإجابة: لا
- Router هو intermediate device مش end device
- Hosts هما الأجهزة اللي بتولد أو بتستقبل البيانات النهائية
- Router بس بينقل البيانات من مكان لمكان
7
في الرسمة اللي فيها 6 nodes، كام منهم Hosts؟
الإجابة:
2 Hosts بس (الأول والأخير)
- الباقي (4 nodes) هما Routers
- Hosts = End devices
- Routers = Intermediate devices
8
هل الـ Switch يعتبر Node؟ ليه؟
الإجابة:
- تقنياً: نعم، لأنه جهاز على الشبكة
- لكن في سياق الـ Slides دي: لا
- السلايدات بتعتبر Node = Host أو Router
- Switch شغال على Layer 2 بس، مش بيعمل routing
- في السياق ده الـ Switch جزء من الـ Link مش Node منفصل
9
ممكن يكون عندنا Node مش Host ومش Router؟ اديني مثال.
الإجابة:
نعم، مثلاً:
- Firewall
- Load Balancer
- Proxy Server
- Gateway
كلهم Nodes لأنهم أجهزة على الشبكة، لكن مش Hosts ولا Routers بالمعنى التقليدي
النقطة المخفية 4: ليه Link مش Connection؟
ما بين السطور:
السلايدات استخدمت كلمة "Link" مش "Connection"

المعنى المخفي:
- Link = الوسط الفيزيائي + البروتوكول
- Connection = علاقة منطقية بين جهازين
- Link ممكن يكون فيه connections كتير
- Link هو المورد المشترك، Connection هو الاستخدام
- في Point-to-Point: link واحد = connection واحد
- في Broadcast: link واحد = connections كتير محتملة
10
إيه الفرق بين Point-to-Point Link و Broadcast Link من ناحية عدد الـ Connections؟
الإجابة:
Point-to-Point:
- Link واحد، Connection واحد ممكن
- بين جهازين بس
- مثال: PPP, HDLC

Broadcast:
- Link واحد، Connections كتير ممكنة
- أي جهاز يقدر يكلم أي جهاز
- مثال: Ethernet, WiFi
11
في Broadcast Link، كل الأجهزة بتشوف كل الـ Frames؟ ده أمان ولا مشكلة؟
الإجابة:
- نعم، كل الأجهزة بتشوف كل الـ Frames
- ده مشكلة أمان محتملة (Sniffing attacks)
- علشان كده محتاجين Encryption على الطبقات الأعلى
- الجهاز المفروض يتجاهل الـ Frames اللي مش ليه
- لكن Hacker ممكن يحط الـ NIC في Promiscuous Mode ويشوف كل حاجة
12
ليه Point-to-Point أسرع من Broadcast رغم إن الاتنين على نفس الـ Medium؟
الإجابة:
- Point-to-Point: مافيش تنافس على الوسط، الجهازين يستخدموه براحتهم
- Broadcast: فيه تنافس، محتاج CSMA/CD أو CSMA/CA
- Broadcast: ممكن يحصل Collisions
- Point-to-Point: مافيش Collisions أصلاً
- Point-to-Point: مافيش Overhead من البروتوكولات المعقدة
النقطة المخفية 5: DLC vs MAC - التقسيم الخفي
ما بين السطور:
السلايد قال "LAN protocols actually use the same strategy"

المعنى المخفي:
- التقسيم لـ DLC و MAC مش مجرد تقسيم نظري
- ده فعلاً موجود في IEEE 802 standards
- DLC = LLC (Logical Link Control) في IEEE
- MAC = MAC Layer في IEEE
- Ethernet مثلاً بيستخدم التقسيم ده فعلاً
- ده معناه إن ممكن يكون عندك نفس الـ LLC على أكتر من MAC
التريكاية الخفية:
السلايد مااتكلمش بوضوح عن إن:
- LLC (802.2) ممكن يشتغل فوق Ethernet (802.3)
- أو فوق Token Ring (802.5)
- أو فوق WiFi (802.11)
- ده معناه إن عندك Flexibility في اختيار الـ MAC من غير ما تغير الـ LLC
13
إيه الفايدة من تقسيم Data Link لـ DLC و MAC؟
الإجابة:
- Modularity: فصل المسؤوليات
- Flexibility: تغيير الـ MAC من غير ما نمس الـ DLC
- Reusability: نفس الـ LLC ينفع مع MACs مختلفة
- Easier maintenance: لو في مشكلة نعرف هي في أنهي طبقة
- Standardization: IEEE قدرت تعمل standards منفصلة لكل واحدة
14
هل كل Data Link Protocols بتستخدم التقسيم لـ DLC و MAC؟
الإجابة:
- لا، مش كلهم
- LANs زي Ethernet و WiFi: نعم
- Point-to-Point protocols زي PPP: لا
- PPP مثلاً مابيحتاجش MAC layer لأن مافيش shared medium
- التقسيم ده مهم بس لما يكون في shared medium
15
DLC بتعمل Flow Control، طب MAC بتعمل إيه بالضبط؟
الإجابة:
MAC Layer مسؤول عن:
- Media Access Control: تنظيم مين يستخدم الوسط امتى
- Framing: تجميع البيانات في Frames
- Addressing: إضافة MAC Addresses
- Error Detection: حساب FCS

DLC Layer مسؤول عن:
- Flow Control: تنظيم سرعة الإرسال
- Error Control: إعادة إرسال الـ Frames الغلط (في بعض البروتوكولات)
- Link Management: إدارة الاتصال
النقطة المخفية 6: ليه MAC Address 48 bits بالضبط؟
ما بين السطور:
السلايد قال "48 bits" لكن ماقالش ليه

المعنى المخفي:
- 48 bits = 6 bytes = 281 trillion عنوان ممكن
- ده عدد كافي لكل جهاز في العالم يكون له عنوان فريد
- أول 24 bits (3 bytes) = OUI (Organizationally Unique Identifier) للشركة المصنعة
- آخر 24 bits = Serial number للجهاز
- IEEE بتبيع الـ OUIs للشركات المصنعة
- الشركة تستخدم الـ 24 bits الباقية لترقيم منتجاتها
تريكاية خطيرة:
السلايد قال "second digit needs to be odd for unicast"
- ده مش الـ digit التاني في العنوان كله!
- ده الـ digit التاني في أول Byte!
- يعني في A3:34:45:11:92:F1
- الـ 3 هو الـ digit التاني
- لو 3 (فردي) = Unicast
- لو 2 (زوجي) = Multicast
16
ليه مااستخدموش 32 bits زي IP Address؟ أو 64 bits؟
الإجابة:
ليه مش 32 bits:
- 32 bits = 4 billion عنوان بس
- ده مش كفاية مع زيادة عدد الأجهزة
- IPv4 نفسها عانت من المشكلة دي

ليه مش 64 bits:
- 48 bits كانت كفاية لما اتعمل Ethernet سنة 1980
- زيادة العدد يزود الـ overhead في كل Frame
- 48 bits = توازن بين العدد الكافي والـ efficiency
- لحد دلوقتي مانفدناش من MAC Addresses
17
حدد نوع كل MAC Address:
A) B2:34:56:78:9A:BC
B) FF:FF:FF:FF:FF:FF
C) 01:00:5E:00:00:01
D) A3:45:67:89:AB:CD
الإجابات:
A) Multicast (B2 - الرقم التاني 2 زوجي)
B) Broadcast (كله Fs)
C) Multicast (01 - الرقم التاني 1 لكن bit الأول 0 و bit التاني 1 = multicast)
D) Unicast (A3 - الرقم التاني 3 فردي)
18
هل ممكن جهازين يكون عندهم نفس الـ MAC Address؟ إيه اللي يحصل؟
الإجابة:
نظرياً: لا، مفروض كل MAC فريد

عملياً: نعم ممكن يحصل:
- أخطاء في التصنيع
- MAC Spoofing (تزوير العنوان)
- Virtual Machines ممكن تولد نفس الـ MAC

اللي يحصل:
- Conflict على الشبكة
- الـ Frames ممكن تروح للجهاز الغلط
- مشاكل في الاتصال
- الـ Switch يتلخبط في الـ MAC Table
النقطة المخفية 7: ليه محتاجين IP و MAC مع بعض؟
ما بين السطور:
السلايد قال "we cannot make a datagram reach its destination using only IP addresses"

المعنى العميق:
- IP Address = عنوان منطقي، بيتغير، hierarchical
- MAC Address = عنوان فيزيائي، ثابت، flat
- IP محتاج للـ routing بين الشبكات
- MAC محتاج للتوصيل المباشر في الشبكة المحلية
- الاتنين مكملين بعض مش بديلين لبعض
السؤال الخادع:
"ليه مانستخدمش IP Address بس في كل حاجة؟"
الإجابة الغلط: "علشان MAC أسرع"
الإجابة الصح: "علشان في الشبكة المحلية محتاجين عنوان فيزيائي يشتغل على Layer 2 من غير ما نحتاج routing، وعلشان نقدر نغير IP من غير ما نغير الهاردوير"
19
إيه اللي يحصل لو حذفنا MAC Addresses واستخدمنا IP بس؟
الإجابة:
مشاكل كتير:
- مش هنقدر نعمل switching على Layer 2
- كل حاجة هتحتاج Router حتى في نفس الشبكة
- Broadcast في الشبكة المحلية مش هيشتغل
- لو غيرت الـ IP هتحتاج تغير الهاردوير
- مافيش separation بين Layer 2 و Layer 3
- التكلفة هتزيد (كل حاجة تحتاج router)
20
إيه اللي يحصل لو حذفنا IP Addresses واستخدمنا MAC بس؟
الإجابة:
كارثة:
- مافيش routing ممكن (MAC is flat, not hierarchical)
- كل Router يحتاج يعرف كل MAC في العالم
- Routing tables هتكون ضخمة جداً
- مافيش subnetting أو network organization
- مش هنقدر ننقل جهاز من شبكة لشبكة بسهولة
- الإنترنت مش هيقدر يشتغل أصلاً
21
ليه IP Address بيتغير لكن MAC Address ثابت؟ إيه الحكمة؟
الإجابة:
MAC Address ثابت:
- مرتبط بالـ Hardware (NIC)
- بيتحرق في الشريحة من المصنع
- يعرف الجهاز فيزيائياً
- مش محتاج يتغير لأنه مش hierarchical

IP Address بيتغير:
- منطقي، مش فيزيائي
- بيعتمد على مكان الجهاز في الشبكة
- لو نقلت الجهاز لشبكة تانية، محتاج IP جديد
- بيساعد في الـ routing والتنظيم
النقطة المخفية 8: ARP Cache والتوقيت
ما بين السطور:
السلايد ماتكلمش عن ARP Cache لكن ذكر إن ARP بيحصل "Anytime"

المعنى المخفي:
- لو كل مرة نحتاج MAC نعمل ARP Request = waste of bandwidth
- علشان كده في حاجة اسمها ARP Cache (ARP Table)
- الجهاز بيحفظ الـ IP-to-MAC mappings
- الـ entries دي بتنتهي صلاحيتها بعد وقت معين (عادة 2-20 دقيقة)
- لو الـ entry موجودة في الـ Cache، مش هنعمل ARP Request تاني
تريكاية مهمة:
سؤال: "كل Frame محتاج ARP Request؟"
- الإجابة: لا!
- أول مرة بس نحتاج ARP
- بعد كده نستخدم الـ Cache
- لو الـ entry expired، نعمل ARP تاني
- ده بيوفر bandwidth كتير جداً
22
Alice عايزة تبعت 1000 Frame لـ Bob. كام ARP Request هتحتاج؟
الإجابة:
واحد بس! (في الأغلب)
- أول Frame: ARP Request
- باقي الـ 999 Frame: من الـ Cache
- إلا لو الـ Cache entry انتهت في النص، وقتها ممكن تحتاج ARP تاني
23
ليه ARP Cache Entries بتنتهي صلاحيتها؟ ليه مانخليهاش دايمة؟
الإجابة:
أسباب مهمة:
- الأجهزة ممكن تطفي أو تسيب الشبكة
- الـ IP Address ممكن يتغير (DHCP)
- الـ NIC ممكن يتغير
- لو entry قديم وغلط هيسبب مشاكل
- Memory management: مش عايزين cache كبير جداً
- Security: منع ARP poisoning attacks من الاستمرار طويل
24
إيه الفرق بين Gratuitous ARP و Normal ARP؟
الإجابة:
Normal ARP:
- بسأل عن MAC address لـ IP معين
- Sender و Target مختلفين

Gratuitous ARP:
- الجهاز بيسأل عن نفسه!
- Sender IP = Target IP
- الهدف: إعلان وجود الجهاز
- أو كشف لو في IP conflict
- أو تحديث ARP caches في الأجهزة التانية
النقطة المخفية 9: ARP Security Issues
ما بين السطور:
السلايد ماتكلمش عن الأمان خالص

الخطر المخفي:
- ARP مافيهوش authentication
- أي جهاز يقدر يبعت ARP Reply حتى لو ماحدش سأله
- ده اسمه ARP Spoofing أو ARP Poisoning
- Hacker يقدر يخلي كل الـ traffic يعدي عليه
- Man-in-the-Middle attack سهل جداً بسبب ARP
التريكاية الخطيرة:
ARP Request = Broadcast
ARP Reply = Unicast
لكن مافيش حاجة تمنع جهاز يبعت Reply بدون ما يستقبل Request!
ده الثغرة الأمنية الأساسية في ARP
25
اشرح كيف يحصل ARP Poisoning Attack بالتفصيل.
الإجابة:
السيناريو:
- Alice (192.168.1.10) عايزة تكلم Bob (192.168.1.20)
- Hacker (192.168.1.15) عايز يتجسس

الخطوات:
1. Hacker يبعت Fake ARP Reply لـ Alice يقول: "Bob's MAC is Hacker's MAC"
2. Alice تحدث الـ ARP Cache بتاعها
3. Hacker يبعت Fake ARP Reply لـ Bob يقول: "Alice's MAC is Hacker's MAC"
4. Bob يحدث الـ ARP Cache بتاعه
5. دلوقتي كل الـ traffic بين Alice و Bob يعدي على Hacker
6. Hacker يشوف كل حاجة ويعمل forward علشان مايكشفش نفسه
26
إزاي نحمي نفسنا من ARP Poisoning؟
الإجابة:
طرق الحماية:
- Static ARP entries: نحط الـ mappings يدوي (مش practical في شبكات كبيرة)
- ARP Inspection: الـ Switch يفحص ARP packets
- Port Security: ربط MAC بـ port معين
- Encryption: استخدام VPN أو HTTPS
- ARP monitoring tools: كشف التغييرات المشبوهة
- Dynamic ARP Inspection (DAI): في managed switches
27
ليه ARP مافيهوش Authentication رغم إنه خطير؟
الإجابة:
- ARP اتصمم زمان (1982) قبل ما الأمان يكون أولوية
- في الوقت ده الشبكات كانت موثوق فيها وصغيرة
- إضافة authentication دلوقتي صعبة (backward compatibility)
- هيزود الـ overhead
- محتاجين نغير كل الأجهزة
- الحل: protocols أحدث زي IPv6 (uses NDP بدل ARP)
النقطة المخفية 10: ليه Target MAC في ARP Request كله أصفار؟
ما بين السطور:
السلايد قال إن Target Hardware Address في Request يكون unknown

التفصيل المخفي:
- مش بس unknown، بيتحط كله 00:00:00:00:00:00
- ده placeholder علشان الـ packet structure تكون ثابتة
- مش ممكن نسيبه فاضي لأن الـ packet له حجم ثابت
- الأصفار معناها "لسه مش عارف"
- بعض الأنظمة بتتجاهل الحقل ده في Request
- لكن في Reply بيتملا بالقيمة الصحيحة
28
ليه مش بنحط FF:FF:FF:FF:FF:FF في Target MAC بدل الأصفار؟
الإجابة:
- FF:FF:FF:FF:FF:FF = Broadcast Address
- لو حطيناها في Target MAC في الـ ARP packet نفسه هيكون confusing
- Broadcast بنستخدمها في الـ Frame Header (Ethernet header)
- لكن في الـ ARP packet نفسه بنحط 00s علشان نقول "مش عارفين"
- ده separation واضح بين "بنبعت broadcast" و "مش عارفين العنوان"
29
الـ Frame اللي حامل ARP Request، الـ Destination MAC فيه إيه؟
الإجابة:
FF:FF:FF:FF:FF:FF (Broadcast)

مهم تفرق:
- Ethernet Frame Header: Dest MAC = FF:FF:FF:FF:FF:FF
- ARP Packet جوه الـ Frame: Target MAC = 00:00:00:00:00:00
- دول مختلفين! واحد في Layer 2 header، والتاني في ARP packet نفسه
النقطة المخفية 11: الفرق بين Same Network و Different Networks
ما بين السطور:
Example 9.4 قال "The two hosts are on the same network"
لكن مثال Alice و Bob كانوا على شبكات مختلفة

الفرق الحاسم:
- Same network: ARP مباشر للجهاز المستهدف
- Different networks: ARP للـ Router (Default Gateway)
- الجهاز بيعرف من الـ Subnet Mask لو نفس الشبكة ولا لأ
- ده decision مهم جداً قبل ما نعمل ARP
التريكاية الكبيرة:
السؤال: "Alice عايزة تبعت لـ Bob، بتعمل ARP لمين؟"
- الإجابة تعتمد على: هل Bob في نفس الشبكة؟
- نفس الشبكة: ARP لـ Bob مباشرة
- شبكة مختلفة: ARP للـ Default Gateway
- كتير من الطلبة بيفتكروا إن ARP دايماً للـ Destination، وده غلط!
30
Alice (192.168.1.10/24) عايزة تبعت لـ Bob (192.168.1.20/24). ARP لمين؟
الإجابة:
ARP لـ Bob مباشرة

السبب:
- Alice: 192.168.1.10
- Bob: 192.168.1.20
- Subnet mask: /24 = 255.255.255.0
- Network portion: 192.168.1
- نفس الشبكة! → ARP مباشر
31
Alice (192.168.1.10/24) عايزة تبعت لـ Bob (192.168.2.20/24). ARP لمين؟
الإجابة:
ARP للـ Default Gateway (Router)

السبب:
- Alice: 192.168.1.x
- Bob: 192.168.2.x
- شبكات مختلفة!
- Alice مش هتقدر توصل Bob مباشرة
- محتاجة Router
- تعمل ARP لـ MAC بتاع الـ Router
32
إزاي الجهاز يعرف يعمل ARP لمين بالضبط؟
الإجابة:
الخطوات:
1. الجهاز عنده Destination IP
2. يعمل AND بين Destination IP والـ Subnet Mask بتاعه
3. يعمل AND بين IP بتاعه والـ Subnet Mask
4. لو النتيجتين زي بعض = Same network → ARP للـ Destination
5. لو مختلفين = Different network → ARP للـ Default Gateway

مثال:
My IP: 192.168.1.10/24
Dest IP: 192.168.1.50
192.168.1.10 AND 255.255.255.0 = 192.168.1.0
192.168.1.50 AND 255.255.255.0 = 192.168.1.0
زي بعض → ARP direct
النقطة المخفية 12: Router بيعمل إيه بالضبط مع الـ Frame؟
ما بين السطور:
الـ Figures بتقول "removes data-link header" و "adds new header"

التفصيل الدقيق:
Router مابيعدلش الـ Frame، بيعمل complete decapsulation وإعادة encapsulation:
1. يستقبل الـ Frame على Interface معين
2. يفحص Destination MAC (لازم يكون ليه)
3. يفحص FCS (Error checking)
4. يشيل الـ Ethernet header كله
5. يطلع IP Packet
6. يشوف Destination IP
7. يدور في Routing table
8. يحدد Next hop
9. يعمل ARP للـ Next hop
10. يحط Ethernet header جديد
11. يحسب FCS جديد
12. يبعت الـ Frame على Interface التاني
33
ليه Router مابيعدلش الـ MAC Address في نفس الـ Frame؟ ليه يشيله كله ويعمله من أول وجديد؟
الإجابة:
أسباب مهمة:
- الـ FCS محسوب على الـ Frame كله، لو غيرنا حاجة لازم نحسبه تاني
- ممكن الـ interface الداخل مختلف عن الخارج (Ethernet → WiFi مثلاً)
- الـ MTU ممكن يختلف، محتاجين fragmentation
- Layer separation: Router شغله Layer 3، فبيتعامل مع Packets مش Frames
- Clean design: كل layer تعمل شغلها كامل
